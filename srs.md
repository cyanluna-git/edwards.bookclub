# 독서동호회 서비스 요구사양서

*작성일: 2026-03-16*
*분석 대상: [독서동호회.pbix](/home/edwards/Dev/edwards.bookclub/독서동호회.pbix)*

## 1. 목적

현재 운영 중인 독서동호회 서비스는 SharePoint List를 데이터 저장소로 사용하고, Power BI 보고서를 통해 회원 현황, 출석 기록, 적립금, 도서 신청 현황을 조회한다. 본 문서는 현행 Power BI 모델과 보고서 구성을 기반으로 현재 서비스가 제공하는 기능과 데이터 구조를 역추적하고, 이를 웹 기반 SQLite 애플리케이션으로 전환하기 위한 요구사양을 정리한다.

## 2. 현행 시스템 요약

### 2.1 데이터 소스

- SharePoint Lists
- Power BI Desktop / Power BI Service 보고서
- Power Query를 통해 SharePoint 테이블을 적재
- DAX 계산 테이블과 Measure를 통해 집계 및 잔액 계산

### 2.2 확인된 Power Query 원본

Power Query 정의상 모든 주요 원본은 동일한 SharePoint 사이트에서 읽는다.

- `Members`
- `도서신청`
- `출석!`
- `출석_DefaultAPI`

참조 원본:
- [독서동호회.pbix](/home/edwards/Dev/edwards.bookclub/독서동호회.pbix)

### 2.3 확인된 보고서 페이지

- `Dashboard`
- `출석부`

### 2.4 확인된 핵심 서비스 기능

현행 보고서와 DAX, 시각화 구성을 기준으로 아래 기능이 실제 운영 기능으로 판단된다.

1. 회원 목록 관리
2. 출석 기록 관리
3. 지역별 출석 집계 조회
4. 월별 출석 추이 조회
5. 회원별 적립금 및 사용 금액 조회
6. 도서 신청 및 구매 이력 관리
7. 추가 납입금 반영
8. 모임 사진 및 후기 조회

## 3. 현행 데이터 모델

### 3.1 테이블 목록

#### `Members`

회원 마스터 테이블.

필드:
- `ID`
- `Title`
- `한글이름`
- `부서`
- `email`
- `Role`
- `Location`
- `isLeader`

해석:
- `Title`: 영문 이름 또는 표시 이름
- `Role`: 정회원, Lead, Lead:총무 등 역할 정보
- `Location`: 지역 또는 소속 거점
- `isLeader`: `Role` 기반 계산 컬럼

관찰:
- 회원 수는 약 52명 수준
- Location 카디널리티는 4개

#### `도서신청`

도서 신청 및 구매/집행 이력 테이블.

필드:
- `ID`
- `Title`
- `작가`
- `출판사`
- `금액`
- `신청인.lookupId`
- `신청인.lookupValue`
- `Progess`
- `표지`
- `링크`
- `신청인한줄평!`
- `별점`
- `신청일`
- `추가납입금`

해석:
- `Title`: 도서명
- `Progess`: 신청 상태 또는 처리 상태
- `금액`: 도서 구매 비용
- `추가납입금`: 회비 외 추가 입금
- `별점`: 도서 후기/평가 성격

관찰:
- 레코드 수는 약 185건 수준
- 신청인과 회원 테이블이 lookup 관계로 연결됨

#### `출석!`

모임 출석 및 활동 기록 테이블.

필드:
- `ID`
- `Title`
- `모임일시`
- `참석자.lookupId`
- `참석자.lookupValue`
- `Location`
- `모임사진.serverRelativeUrl`
- `모임사진.serverUrl`
- `No적립금?`
- `ImageURL`
- `후기`

해석:
- 한 출석 레코드는 특정 모임과 특정 참석자 연결 행으로 해석됨
- `No적립금?`는 출석은 했지만 적립 대상이 아닌 예외 처리 플래그
- `ImageURL`은 Power Query에서 사진 URL을 조합해 생성한 계산 컬럼
- `후기`는 모임 회고 또는 기록 텍스트

관찰:
- 레코드 수는 약 149건 수준
- 참석 회원 수는 약 40명 수준
- 지역 카디널리티는 4개

#### `출석_DefaultAPI`

SharePoint 기본 API 응답을 별도 로드한 테이블.

용도 추정:
- `출석!`에서 직접 펼치기 어려운 메타데이터 또는 후기 등 원본 보강
- `Title` 기준으로 `출석!`와 연결됨

현재 웹 전환 시 판단:
- 운영 핵심 테이블이 아니라 SharePoint API 제약을 우회하기 위한 보조 수집 테이블일 가능성이 높음
- 웹 전환 후에는 별도 유지 필요성이 낮음

#### `Calendar`

DAX 계산 테이블.

생성 범위:
- `2023-11-01`부터 `2026-12-31`까지 날짜 생성

생성 컬럼:
- `Date`
- `Year`
- `Month`
- `MonthNO`
- `YearMonth`
- `WeekNO`
- `WeekDay`

용도:
- 월별 필터
- 상대 날짜 필터
- 월별 집계 시계열 축

#### `Reserves`

DAX 계산 테이블.

정의:
- `Reserves = 'Members'`

해석:
- 별도 물리 테이블이 아니라 적립금 피벗 표시를 위한 표시용 계산 테이블
- 웹 전환 시 독립 테이블로 구현할 필요는 없고 회원별 집계 뷰 또는 SQL 쿼리로 대체 가능

### 3.2 관계

확인된 관계:

1. `도서신청.신청인.lookupId -> Members.ID`
2. `출석!.참석자.lookupId -> Members.ID`
3. `출석!.Title -> 출석_DefaultAPI.Title`
4. `도서신청.신청일 -> Calendar.Date`
5. `출석!.모임일시 -> Calendar.Date`

웹 전환 시 핵심 관계:

1. 회원 1:N 도서신청
2. 회원 1:N 출석
3. 일정/모임 1:N 출석참여
4. 모임 1:N 사진
5. 모임 1:1 또는 1:N 후기

## 4. 현행 비즈니스 규칙

### 4.1 적립금 규칙

Measure 정의를 기준으로 확인된 규칙:

- 일반 회원 출석 적립: `5000`
- 리더 출석 적립: `10000`
- 적립 제외: `출석!.No적립금? = TRUE`
- 총 적립금 = 적립 대상 출석 수 x 역할별 포인트
- 잔액 = 적립금 - 도서 구매 금액 + 추가 납입금

### 4.2 도서 구매 집행 규칙

- 도서 구매 합계는 `신청일 > 2026-01-01` 기준으로 계산됨
- 이는 회계 기수 또는 운영 연도 경계가 존재함을 의미
- 웹 전환 시 운영 연도 설정을 하드코딩하지 말고 관리 가능한 설정값으로 분리해야 함

### 4.3 출석/운영 화면 규칙

`출석부` 페이지 기준:

- 최근 기간 또는 상대 날짜 기준으로 조회
- `YearMonth` 드롭다운으로 월 선택
- 지역별 참석자 수 카드 제공: 전체, 천안, 아산, 분당, 동탄
- 사진이 있는 기록과 사진이 없는 기록이 분리되어 보일 수 있음
- 모임 사진과 참석자 명단을 함께 조회함

## 5. 기능 요구사항

### 5.1 회원 관리

1. 운영자는 회원을 등록, 수정, 비활성화할 수 있어야 한다.
2. 회원은 이름, 한글이름, 부서, 이메일, 역할, 지역 정보를 가져야 한다.
3. 회원 역할은 일반회원과 운영진을 구분할 수 있어야 한다.
4. 역할 변경 시 적립금 계산 기준이 즉시 반영되어야 한다.

### 5.2 모임 관리

1. 운영자는 모임 일시와 지역을 등록할 수 있어야 한다.
2. 모임별 사진과 후기를 저장할 수 있어야 한다.
3. 모임은 월 단위로 조회 가능해야 한다.
4. 모임은 운영 연도 또는 기간 기준으로 필터링 가능해야 한다.

### 5.3 출석 관리

1. 운영자는 모임별 참석자를 다중 선택으로 기록할 수 있어야 한다.
2. 출석 기록에는 적립 제외 여부를 저장할 수 있어야 한다.
3. 출석은 회원 단위, 지역 단위, 월 단위로 집계 가능해야 한다.
4. 특정 모임의 참석자 목록을 즉시 조회할 수 있어야 한다.

### 5.4 적립금/회비 관리

1. 시스템은 회원별 누적 출석 적립금을 계산해야 한다.
2. 시스템은 회원별 역할에 따라 포인트 단가를 다르게 적용해야 한다.
3. 시스템은 적립 제외 출석을 계산에서 제외해야 한다.
4. 시스템은 도서 구매 금액과 추가 납입금을 반영해 잔액을 계산해야 한다.
5. 운영자는 운영 연도 기준 시작일을 변경할 수 있어야 한다.

### 5.5 도서 신청 관리

1. 회원은 도서를 신청할 수 있어야 한다.
2. 도서 신청에는 도서명, 작가, 출판사, 금액, 링크, 후기, 별점, 상태를 저장할 수 있어야 한다.
3. 운영자는 신청 상태를 변경할 수 있어야 한다.
4. 도서 신청 내역은 신청자별, 기간별, 상태별로 조회 가능해야 한다.

### 5.6 사진/후기 관리

1. 모임 사진은 모임 단위로 업로드 및 조회 가능해야 한다.
2. 모임 후기는 텍스트로 저장 가능해야 한다.
3. 월별 모임 사진 갤러리 화면이 제공되어야 한다.

### 5.7 대시보드/리포트

1. 월별 출석 추이 차트가 제공되어야 한다.
2. 지역별 출석 비교 차트가 제공되어야 한다.
3. 회원별 적립금, 도서 사용 금액, 잔액 테이블이 제공되어야 한다.
4. 월별 출석 현황 페이지가 제공되어야 한다.
5. 대시보드는 관리자용 기준으로 설계되어야 한다.

## 6. 비기능 요구사항

1. 단일 서버 또는 단일 배포 단위에서 운영 가능해야 한다.
2. 초기 버전은 SQLite 기반으로 운영 가능해야 한다.
3. 데이터 모델은 향후 PostgreSQL로 이전 가능해야 한다.
4. 관리 화면에서 데이터 수정이 쉬워야 한다.
5. 사진 저장은 로컬 파일 또는 S3 호환 저장소로 전환 가능해야 한다.
6. Power BI 없이도 동일한 핵심 운영 지표를 웹 UI에서 조회 가능해야 한다.

## 7. 데이터 마이그레이션 요구사항

1. SharePoint List의 현재 데이터를 일괄 추출할 수 있어야 한다.
2. 회원, 출석, 도서신청, 사진/후기 데이터를 정규화된 구조로 적재해야 한다.
3. 기존 Power BI 계산 결과와 웹 애플리케이션 계산 결과를 비교 검증해야 한다.
4. 기존 `출석_DefaultAPI` 같은 SharePoint 종속 보조 구조는 목표 시스템에서 제거해야 한다.

## 8. 오픈 이슈

1. `Progess`의 실제 상태값 정의가 문서상 명확하지 않다.
2. `Title`이 `출석!`에서 모임 식별자 역할인지 화면용 제목인지 추가 확인이 필요하다.
3. 사진이 모임 1건당 1장인지 여러 장인지 현행 모델만으로는 확정하기 어렵다.
4. `후기`가 모임 단위인지 참석 기록 단위인지 추가 확인이 필요하다.
5. 권한 체계는 Power BI 보고서만으로는 충분히 확인되지 않았다.

## 9. 결론

현행 시스템은 단순 보고서가 아니라, 실질적으로 다음 업무를 수행하는 경량 운영 플랫폼이다.

- 회원 관리
- 모임 및 출석 관리
- 적립금 회계 관리
- 도서 신청 및 구매 관리
- 모임 사진 및 후기 기록

따라서 웹 전환 시 단순 대시보드 재구현이 아니라, 운영 CRUD와 집계 리포트를 모두 포함한 관리형 업무 시스템으로 정의해야 한다.
